# Draft Channel Rules
# 在 Step 8d 起草時讀取此檔案。

## 草稿檔案命名規則

格式：`{output_dir}/drafts/{NN}-{channel}-{recipient}.md`

- `NN` = 對應 dispatch 清單編號，補零（01、02 …）
- 外發類：`02-line-to-elmer.md`、`04-email-to-client.md`
- 內部類：`03-validation-agenda-internal.md`、`07-poc-notes-internal.md`

## 草稿檔案格式（所有管道通用）

```markdown
<!-- 發送對象：XXX / 管道：LINE / 日期：YYYY-MM-DD -->
<!-- 複製下方虛線內的文字 -->

---

[正文]

---
```

- **外發管道（LINE、Email、Slack）**：`---` 之間只放純文字或符合該管道的格式，不放任何 Markdown 語法（`*`、`#`、`>`等）
- **內部文件（議程、備忘、Executive Brief）**：可用完整 Markdown 結構

---

## 各管道規則

| 管道 | 正文格式 | 資訊邊界 | 長度 |
|------|---------|---------|------|
| **LINE / 非正式外部** | 純文字、口語化、emoji OK | 去除費用數字、合約細節、內部討論 | 5-15 行 |
| **Slack / 內部群組** | Bullet points、可用 `*bold*` | 去除對外費用、法律字眼 | 20-30 行 |
| **Email（外部客戶）** | Subject + Body + 附件提醒 | 去除內部討論、費用（除非是報價）、法律字眼；偵測對方語言自動切換 | 200-400 字 |
| **Email（法務 / 合規）** | Subject + 條款引用 + 問題清單 | 只含相關條款，**不做法律判斷** | 依需求 |
| **Executive Brief** | 背景 → 摘要 → 需要的決定 → 建議 | 只含決策者需要的資訊，不含執行細節 | 1 頁以內 |
| **Calendar 邀約** | 會議名稱 + 議程大綱 + 參與者 | 自動建議時段（如會議中有提到） | 簡短 |
| **任務指派** | 任務描述 + 期限 + 背景 context | 附上 meeting-notes.md 相關段落引用 | 依任務 |
| **議程 / Demo Script** | Markdown 結構 + 備用 Q&A + Checklist | 僅內部使用 | 依需求 |
| **研究 / 備忘** | 結構化筆記 + 關鍵問題 + 搜尋起點 | 僅內部使用 | 依需求 |

## 敏感內容處理規則

起草前先確認：

1. **費用數字** — 只在正式報價 Email 或內部文件中出現，不進 LINE / Slack
2. **第三方客戶資訊** — 不在任何外發草稿中提及其他客戶的名稱、績效、內部問題
3. **法律 / 合規議題** — 不在一般 Email 中提及，升級到 Email（法務）管道
4. **人事 / 薪資** — 永遠不進任何外發草稿

發現敏感內容時，在草稿頂部 HTML 注釋加標記：
```
<!-- ⚠️ 敏感提醒：本草稿含 [項目]，請確認後再發送 -->
```

## 中途調整規則

使用者在任何時候可以用自然語言調整，不需要特殊指令：

- 「語氣太正式了」→ 改語氣
- 「#1 跟 #3 合併」→ 合併成單一草稿
- 「也幫我做 #6」→ 追加起草
- 「這個先不用」→ 跳過，繼續下一份
- 「改成英文」→ 重新生成對應語言版本
